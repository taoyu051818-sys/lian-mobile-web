# 编辑原则

这份文件记录“猴岛情报站 / 黎安信息流”的内容处理原则。以后整理 NodeBB 帖子、推荐流、小程序或移动端信息卡片时，都按这里执行。

## 读者

默认读者是黎安国际教育创新试验区的学生。

写作时先问：学生为什么要点开、现在能拿走什么、和他的校园生活有什么关系。不要写给管理者看，不要写成通知复述，不要写成资料库说明。

## 信息处理

- 先读全文，再编辑；不要只靠标题、程序规则或关键词拼接摘要。
- 低置信度的信息不强行关联地点、时间或推荐理由。
- 时效信息单独维护，不用 NodeBB 编辑时间代替活动时间。
- 原文链接保留在结构化字段里，例如 `post-metadata.json` 的 `sourceUrl`；正文里不直接裸露 URL。
- 来源不是微信公众号时，统一用“跳转到原文”，不要写“公众号原文”。

## 写作风格

- 用学生能自然读下去的语言，避免“学生关注：”“内容：”“时间上可以先记一下：”这类模板腔。
- 开头先给判断或场景，不先堆背景。
- 分段要短，重点要清楚；适合移动端阅读。
- 可以有小标题，但小标题要像人在整理信息，不像公文目录。
- 不夸大，不硬推，不把普通信息包装成重大事件。

## 图文帖子

- NodeBB 正文使用 HTML，不依赖 Markdown 渲染。
- 图片先上传 Cloudinary，正文使用 `<figure><img ... /></figure>`。
- 正文图片 URL 使用 Cloudinary CDN，不使用本地路径。
- 抓取过程中的截图、原图、页面缓存、临时 JSON 和发布脚本不进 Git，不部署到服务器。

## 推荐流

- 推荐流按“批次”设计，而不是只按时间倒序。
- 每一批是一版人工/LLM 排过的队列，例如 `[93, 94, 95, 96, 97]`。
- 用户侧可以在批次内部做轻微个性化：已读放到本批队尾，未读靠前。
- 推荐理由主要看时效性、重要性、学生可行动性、校园相关度和内容质量。
- 每天帖子数量少时，优先维护一版高质量推荐流，不追求无限流数量。

## 文件边界

- `data/feed-rules.json`：推荐流批次、标签权重和置顶队列。
- `data/post-metadata.json`：时间、优先级、原文链接等结构化元数据。
- `outputs/club-posts/*.md`：可保留已编辑 HTML 正文草稿。
- `outputs/club-posts/images/`、`public/snapshot-assets/`、`data/nodebb-snapshot.json`：运行时或抓取产物，不进 Git。

## 发布前检查

- 正文是否像学生写给学生看的内容。
- 是否已经使用 HTML，而不是 Markdown 图片语法。
- 图片是否能通过 Cloudinary 访问。
- 正文是否没有裸露原文 URL。
- `sourceUrl` 是否指向正确原文。
- 卡片摘要是否不会显示图片代码或模板废话。
