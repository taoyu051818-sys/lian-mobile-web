# Task: Map v2 Render Workflow

## Goal

Define the workflow that turns editable map data into a richer visual base map through an external renderer or AI tool.

LIAN does not implement the renderer. LIAN owns the structured input, versioning, upload/approval flow, and frontend binding of the rendered output.

## Product Scope

The admin tool should let an operator:

1. edit map layers in structured form;
2. export a render specification;
3. send the spec to an external AI/design/rendering workflow;
4. upload or reference the rendered output image;
5. preview it under existing interactive overlays;
6. approve and publish a new map version.

## Render Job Model

```json
{
  "id": "render-20260502-001",
  "mapVersion": "map-v2.1",
  "status": "draft",
  "inputSpecPath": "",
  "outputImageUrl": "",
  "outputImageSize": {
    "width": 4096,
    "height": 4096
  },
  "prompt": "",
  "renderer": "external_ai",
  "createdAt": "",
  "approvedAt": "",
  "publishedAt": "",
  "notes": ""
}
```

Allowed statuses:

```text
draft
exported
rendered
reviewing
approved
published
rejected
archived
```

## Render Spec Contents

The exported spec should include:

- campus bounds;
- current base-layer asset references;
- natural environment layers;
- roads and paths, including road type, points, junctions, segments, and render hints;
- shuttle routes;
- building footprints;
- building groups;
- entrances;
- icon placements;
- labels/cards;
- render hints per object;
- desired output dimensions;
- safe-area notes for overlay readability.

Road rendering requirements:

- `points[]` remain the geometry source of truth.
- `junctions[]` are generated by the road engine from line crossings, endpoint snaps, and near-segment connections.
- `segments[]` are generated by splitting roads at detected junctions.
- `renderHint` tells the external renderer whether the road should look like asphalt, a paved walking path, a shuttle overlay, a service path, etc.
- Curves and corners are classified automatically from line geometry and exported as hints.
- Curves may be rendered visually, but the returned output must stay aligned to the original geometry.
- External renderers should not invent new intersections or reroute roads; they should style the topology exported by LIAN.

External graphic assets:

- Render specs may reference imported building/environment/icon assets.
- Assets should be bound to structured objects through `boundObjectType` and `boundObjectId`.
- External renderers may create richer visual versions, but LIAN keeps the structured object as the source of click behavior and future editing.
- If a rendered graphic replaces an existing asset, the object id must stay stable.

The spec should not include:

- API keys;
- user private data;
- unpublished private posts;
- raw auth/session data.

## Frontend Binding

The published render output becomes a base visual asset. Interactive overlays remain separate and editable.

Frontend stack:

```text
rendered image/base layer
  -> Gaode or coordinate reference layer, if enabled
  -> LIAN roads/areas/buildings overlay
  -> clickable icons
  -> post cards
```

Do not flatten clickable icons, buildings, or posts into the rendered image. They must stay interactive overlay data.

## Backend Requirements

Expected admin APIs:

```text
POST /api/admin/map-v2/render-jobs
GET  /api/admin/map-v2/render-jobs
GET  /api/admin/map-v2/render-jobs/:id
POST /api/admin/map-v2/render-jobs/:id/export
POST /api/admin/map-v2/render-jobs/:id/output
POST /api/admin/map-v2/render-jobs/:id/approve
POST /api/admin/map-v2/render-jobs/:id/publish
```

## Non-Goals

- Do not implement image generation in LIAN.
- Do not call external AI services automatically in the first version.
- Do not remove the editable structured layers after rendering.
- Do not make rendered output the only source of truth.

## Validation

- render spec exports valid JSON;
- render spec excludes secrets and private user data;
- uploaded output can be previewed with existing overlays;
- rollback to previous map version is documented and possible.
