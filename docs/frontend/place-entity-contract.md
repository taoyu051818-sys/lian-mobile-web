# Place Entity Contract

> Status: draft | Source: #146, #132, #137, #141, #145

Defines the frontend contract for place entities: data shapes, ID semantics, status presentation, correction/save flows, AI summary display, recent posts visibility, freshness/cache, and cross-view consistency.

## 1. PlacePreview vs PlaceSheet

Two layers exist for place data at different depths.

### PlacePreview (lightweight)

Used by map markers, feed cards, and search results. Carries enough to render a pin, chip, or card without a full fetch.

```ts
// src/types/place.ts — PlaceRef
interface PlaceRef {
  id: string;
  name: string;
  type?: string;
  status?: PlaceStatus;
}
```

- Consumed directly from `MapLocation.place` and `FeedItem.placeLabel`.
- Does not include summary, stats, recentPosts, or coordinates.
- `MapLocation.id` is the map marker ID; `MapLocation.placeId` or `MapLocation.place.id` is the canonical place ID. They may differ (see section 4).

### PlaceSheet (full detail)

Used by the place detail panel. Fetched via `fetchPlaceSheet(id)`.

```ts
// src/types/place.ts
interface PlaceSheet {
  id: string;
  name: string;
  type?: string;
  lat?: number;
  lng?: number;
  status: PlaceStatus;
  updatedAt?: string;
  source?: SourceSignal;
  summary?: PlaceSummary;
  stats?: PlaceStats;
  recentPosts?: PlaceRecentPost[];
}
```

- Opened from PostDetailPanel or as a future "view place details" action on MapLeafletView.
- MapLeafletView currently only uses `PlacePreview` fields. To show full place info, it should load a `PlaceSheet` on demand.

**Rule:** Components must not duplicate PlaceSheet fields from PlacePreview. If detail-level data is needed, fetch the sheet.

## 2. Place Status: Label, Description, and Tone

`PlaceStatus` has six values. All views must use a single presenter (`presentPlaceStatus`) — no local hardcoded mappings.

| Status | Label | Tone | Description (user-facing) |
|---|---|---|---|
| `confirmed` | Confirmed | `success` | Verified by multiple sources |
| `pending` | Pending | `warning` | Awaiting verification; info may be incomplete |
| `disputed` | Disputed | `warning` | Conflicting reports; info may be unreliable |
| `expired` | Expired | `muted` | May no longer be current; check before visiting |
| `ai-organized` | AI-organized | `info` | Summarized by AI from posts; not human-verified |
| `official` | Official | `success` | Provided by the institution |
| *(unknown)* | Unknown | `muted` | Status unavailable |

**Presenter contract:**

```ts
interface PlaceStatusPresentation {
  label: string;
  tone: "success" | "warning" | "info" | "muted";
  description: string;
}
```

- Map `TrustBadge` must consume `tone` from the presenter, not re-derive from raw status.
- Feed card, detail panel, map popup, and search results all call the same presenter.
- Unknown/unrecognized statuses fall back to `muted` tone with "Unknown" label and emit a diagnostics event.

## 3. Correction, Save, Recent Posts, AI Summary, Freshness, Cache

### Correction

`PlaceStats.correctionCount` is read-only today. The contract for write operations:

```ts
interface PlaceCorrectionPayload {
  name?: string;
  type?: string;
  location?: { lat: number; lng: number };
  status?: PlaceStatus;
  summary?: string;
  reason: string;
}
```

- `submitPlaceCorrection(placeId: string, payload: PlaceCorrectionPayload): Promise<void>`
- Correction is distinct from content reporting (#129). It targets place metadata, not post content.
- UI entry points: PlaceSheet detail panel and (future) map place popup.
- After submission, show a local "submitted" toast; do not optimistic-update the sheet.

### Save / Bookmark

`PlaceStats.savedCount` is read-only today. Contract for write operations:

- `togglePlaceSave(placeId: string, nextSaved: boolean): Promise<void>`
- "Saved places" is a distinct entity type from "saved posts" in the profile.
- If MVP does not ship place save, hide `savedCount` from the UI rather than showing a dead number.

### Recent Posts

`PlaceRecentPost` uses `FeedItemId | string` for `tid`. Contract:

- Each recent post must be navigable to the post detail view (or explicitly marked read-only).
- Actor display uses the centralized actor presenter (#145) — no inline fallback logic.
- Visibility/deleted/restricted handling (see section 5).

### AI Summary

`PlaceSummary` fields:

| Field | Meaning | UI requirement |
|---|---|---|
| `text` | Summary content | Display as primary text |
| `aiGenerated` | Whether AI-generated | Show "AI-organized" badge when `true` |
| `confidenceLabel` | Confidence level | Show alongside badge for low-confidence |
| `sourceCount` | Number of source posts | Show as "Based on N posts" meta |

- When `aiGenerated` is true, display a disclaimer: "This summary was generated by AI from community posts and may be incomplete."
- Low confidence or disputed/expired places should show a more cautious variant: "This AI summary may be inaccurate. Help improve it."
- Provide a "This summary is inaccurate" feedback link (distinct from place correction).

### Freshness

`PlaceSheet.updatedAt` indicates last data update.

- Display "Updated X ago" in the detail panel.
- For `pending` or `expired` status, append a caution hint: "This info may be outdated."
- After a correction submission, re-fetch the PlaceSheet to get the latest `updatedAt`.

### Cache

- PWA may cache public PlaceSheet data. Sensitive/visibility-restricted content must not be cached offline.
- Cache key includes `placeId` and `updatedAt` — stale entries are invalidated when `updatedAt` changes.
- Map data cache (#132) uses `dataVersion`/`schemaVersion`, not PlaceSheet-level cache.

## 4. ID Relations: locationId / placeId / place.id

Three distinct ID concepts exist:

| ID | Defined in | Meaning | Type |
|---|---|---|---|
| `MapLocation.id` | `src/types/map.ts` | Map marker identity | `string` |
| `MapLocation.placeId` | `src/types/map.ts` | Canonical place ID the marker references | `string?` |
| `MapLocation.place.id` | `src/types/place.ts` (PlaceRef) | Same canonical place ID, carried as a ref | `string` |
| `PlaceSheet.id` | `src/types/place.ts` | The canonical place entity ID | `string` |
| `PublishLocationDraft.locationId` | `src/types/publish.ts` | Map marker ID from the publish flow | `string?` |
| `PublishLocationDraft.placeId` | `src/types/publish.ts` | Canonical place ID from the publish flow | `string?` |

**Rules:**

1. **Canonical place ID** (`PlaceSheet.id`, `PlaceRef.id`, `MapLocation.placeId`) is the stable identity used for fetching sheets, saving, correction, and deduplication. Generated by the backend.
2. **Map marker ID** (`MapLocation.id`) is an ephemeral rendering identity. It must not be used to fetch a PlaceSheet or as a cache key for place data.
3. When both `MapLocation.placeId` and `MapLocation.place?.id` are present, they must agree. If they differ, prefer `placeId` and log a diagnostics warning.
4. If `MapLocation.placeId` is absent, the marker has no linked canonical place. The UI should not offer place detail, correction, or save actions.
5. In the publish flow, the backend resolves `locationId` to a canonical `placeId`. The frontend should not assume they are the same.
6. All place IDs are opaque strings. No numeric sorting or comparison (#137).

## 5. Visibility, Deleted, Restricted: Recent Posts Fallback

`PlaceRecentPost` entries may reference posts that are deleted, restricted, or visibility-filtered. The contract:

| Post state | `recentPosts` display | Fields shown |
|---|---|---|
| Visible, accessible | Normal card with title, excerpt, actor, time | All |
| Deleted | Placeholder: "This post has been deleted" | None — no title, excerpt, actor, or image |
| Restricted / visibility-filtered | Placeholder: "This content is not available" | None — no title, excerpt, actor, or image |
| Missing `tid` | Skip the entry entirely | N/A |

**Rules:**

- Never leak title, excerpt, or actor info for restricted/deleted posts. Use generic placeholders only.
- Placeholder cards use `muted` tone and are not tappable.
- Actor display on visible posts uses the centralized actor presenter (#145), not inline fallback.
- recentPosts should be capped at 3 entries. If all 3 are deleted/restricted, hide the "Recent posts" section entirely.
- The `tid` field must be a normalized `PostId` (string) after adapter processing (#137). If the raw value is numeric, convert in the adapter, not in the component.

## 6. Cross-View Consistency

| Concern | Map view | Detail view | Feed card | Search |
|---|---|---|---|---|
| Status badge | `presentPlaceStatus` | `presentPlaceStatus` | `presentPlaceStatus` | `presentPlaceStatus` |
| Place name | From PlacePreview | From PlaceSheet | From FeedItem | From search result |
| Summary | Not shown (preview only) | Full with AI badge | Not shown | Not shown |
| Correction | Future map popup | PlaceSheet panel | N/A | N/A |
| Save | Future map popup | PlaceSheet panel | N/A | N/A |
| Recent posts | Not shown | Shown with visibility rules | N/A | N/A |
| Actor display | Actor presenter | Actor presenter | Actor presenter | Actor presenter |

**Key invariant:** The same place shown in different views must display the same status label, tone, and name. If PlacePreview carries stale data, the detail view (PlaceSheet) is authoritative on open.

## 8. Test Fixtures

Place fixtures should cover:

- All six `PlaceStatus` values plus an unknown/invalid status
- AI summary: high confidence, low confidence, no summary
- Stats: with counts, all zero, missing stats object
- Recent posts: all visible, mixed with deleted/restricted, all deleted, empty array, missing `tid`
- Correction: success, error, rate-limited
- Save: success, error
- ID scenarios: `placeId` present/absent, `placeId` differs from `place.id`, marker ID vs place ID
