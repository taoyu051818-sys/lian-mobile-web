# LIAN Campus UI / UX Guidelines V0.1

LIAN 不是论坛加地图，而是以地点为归宿的校园信息系统。

卡片是信息的容器，马甲是用户的身份，地点是内容的归宿。

视觉让它统一，动效让它连贯，状态让它可信，身份让它安全，地点让它沉淀。

## 1. 产品设计原则

1. 信息优先于装饰。
2. 信息流优先于功能宫格。
3. 卡片承载信息。
4. 马甲承载身份。
5. 地点组织沉淀。
6. 算法调节内容生态。
7. 玻璃只做浮层材质。
8. AI 只做整理和辅助，不做事实权威。
9. 主操作永远比次操作明显。
10. 内容要可搜索、可归档、可复用。

LIAN 的 UI 判断优先级是：内容 > 身份 > 地点 > 来源与状态 > 操作 > 装饰。

## 2. 视觉基础

LIAN 的视觉方向是 Campus Utility Glass：校园生活工具感 + 滨海轻玻璃感。

Utility 是核心，Glass 是材质。工具性优先，玻璃感辅助。

### 颜色

- 背景：米白、浅暖灰。
- 主色：蓝绿色，用于主操作、当前选中、品牌识别。
- 海岸蓝：地图、地点、空间信息。
- 草地绿：已确认、贡献、自然生活类。
- 黄色 / 橙色：待确认、AI 整理、交易或风险提醒。
- 红色：删除、举报、错误。
- 深墨色：主要文字。
- 灰色：辅助信息、时间、来源。

颜色可以辅助识别，但不能成为唯一语义。

### 字体与排版

使用系统中文字体。标题短，正文不挤。卡片里最多突出一个主信息。

推荐字号：

| 类型 | 字号 |
| --- | --- |
| 页面标题 | 22–24px |
| 模块标题 | 18–20px |
| 卡片标题 | 16px |
| 正文 | 14px |
| 辅助信息 | 12px |
| 标签 | 11–12px |

### 间距、圆角、阴影

使用 4px 栅格：4 / 8 / 12 / 16 / 24 / 32。

| 元素 | 圆角 |
| --- | --- |
| 标签 Chip | 999px |
| 按钮 | 12px |
| 普通卡片 | 16px |
| Sheet / 大浮层 | 24px |
| Orb 图标按钮 | 999px |

阴影只保留两级：卡片阴影和浮层阴影。

## 3. 玻璃 UI 与层级

玻璃不是风格主角，玻璃是层级工具。

| 类型 | 用途 | 规则 |
| --- | --- | --- |
| Glass Bar | 顶部栏、底部导航 | 轻 blur，低阴影 |
| Glass Panel | 地图详情卡、发布表单、底部抽屉 | 中 blur，中阴影 |
| Glass Orb | 返回、关闭、定位、发布悬浮按钮 | 圆形，识别强 |
| Tonal Pill | 保存草稿、重新生成、筛选 | 浅色、低强调 |
| Filled Pill | 发布、确认、下一步、提交 | 实色、高强调 |

决定流程结果的按钮，不能只是玻璃按钮。

必须使用 Filled Pill 的操作：发布、提交、确认、下一步、保存修改、确认拼车、确认交易。

可以使用玻璃或浅色按钮的操作：返回、关闭、定位、筛选、保存草稿、重新生成、展开更多。

玻璃 UI 禁止大面积覆盖正文、长列表每项、评论正文和复杂表单底层。

## 4. Card System 卡片系统

统一卡片底色，轻量类型色，结构优先，颜色辅助。

LIAN 信息流不使用大面积类型色。卡片整体保持统一底色。内容类型通过 Type Chip、Icon、轻色条、来源状态和内容模块区分。

### 卡片统一结构

```txt
[Type Chip] [Location Chip] [Trust Badge]

标题

摘要 / 图片 / 特定内容模块

IdentityBadge · ContributionBadge
时间 · 评论 · 收藏 · 沉淀状态
```

### 卡片母类型

V0.1 先收敛成 4 个母卡片：

| 母卡片 | 覆盖内容 |
| --- | --- |
| ContentCard | 帖子、经验、讨论、通知、热帖 |
| PlaceCard | 地点、楼栋、饭堂、商家聚合 |
| ActionCard | 发布、纠错、补充、贡献记录 |
| SummaryCard | AI 摘要、系统整理、资料库总结 |

### 类型色规则

类型色只用于小面积 Type Chip、细线、图标或 badge，不用于整张卡片背景。

显性标识数量限制：

- Type Chip：最多 1 个。
- Location Chip：最多 1 个。
- Trust Badge：最多 1 个。
- Tag Chip：最多 2–3 个，更多折叠。
- Identity / Contribution：最多 2 个。

## 5. 身份、贡献标签与 Tag Chip

### 身份系统

LIAN 的身份系统包含展示身份和信誉身份。

```txt
小蓝鲸 · 马甲身份
饭堂观察员 · 菜单纠错者
```

展示规则：

- 卡片上最多展示 1 个身份标签 + 1 个贡献标签。
- 详情页可以展示更多身份信息。
- 个人页展示完整身份和贡献历史。
- 发布页必须显示当前发布身份。
- 评论框必须显示当前评论身份。
- 拼车和闲置必须进入实名或半实名确认。

发布页必须显示：你将以「小蓝鲸 · 马甲身份」发布。

评论框显示：以「小蓝鲸」评论。

### 标签 Chip

标签是内容连接系统，不是装饰。

规则：

- 展示时必须带 #。
- 用户输入时 # 可选。
- 系统保存时统一补 #。
- 最多 5 个。
- 单个标签正文最多 15 个字符。
- 允许中文、英文、数字、下划线、连字符。
- 不允许空格、表情、特殊符号。
- 输入后必须转成 chip。

标签分三类：

| 类型 | 示例 | 来源 |
| --- | --- | --- |
| 内容标签 | #避坑 #晚饭 #求助 | 用户输入 |
| 地点标签 | #三食堂 #宿舍区 | 地点绑定自动生成 |
| 系统标签 | #AI整理 #官方通知 #已验证 | 系统生成，用户不可伪造 |

## 6. 信息流、地点与地图沉淀

信息流负责分发，地图负责定位，地点页负责沉淀。

首页不是功能宫格优先，而是信息流优先。

地图不是首页主舞台。地图是空间索引。

内容生命周期：

```txt
内容 → 信息流分发 → 地点归档 → 被搜索 / 地图 / AI 复用
```

每条内容发布时应尽量回答：

- 它能不能绑定地点？
- 它能不能进入地点页？
- 它能不能变成长期资料？

地图默认展示稳定地点、当前筛选、附近高相关、少量热帖和用户正在操作的内容。不要默认展示所有帖子。

## 7. 动效、交互、状态与反馈

### 动效

动效必须解释关系，而不是制造热闹。

时长建议：

| 类型 | 时长 |
| --- | --- |
| 微反馈 | 100–180ms |
| 卡片展开 | 180–240ms |
| 页面转场 | 240–320ms |
| 地图 Sheet | 250–350ms |

规则：

- 页面转场只用 transform / opacity。
- 不动画 width / height / top / left。
- 长列表不要每次大面积 reveal。
- 玻璃 blur 不参与强动画。
- 必须支持 prefers-reduced-motion。

### 交互

事件优先级：按钮 > chip > 图片 > 身份 > 卡片主体。

默认规则：

- 点击卡片主体：进入详情。
- 点击地点 chip：进入地点页。
- 点击标签 chip：进入标签页或搜索结果。
- 点击身份：进入用户主页或身份说明。
- 点击图片：打开 lightbox。
- 长按卡片：更多操作。
- 下拉首页：刷新。
- 上滑地点 sheet：展开。

### 状态

每个组件至少考虑：默认、按下、选中、禁用、加载中、成功、失败。

每条可沉淀信息至少考虑：已确认、待确认、有争议、已过期、AI 整理、官方信息。

### 反馈

所有写操作必须有即时反馈。失败时必须告诉用户下一步。

示例：

- 发布成功，已关联到「三食堂」。
- 发布失败，草稿已保存，可以稍后重试。
- 已提交纠错，确认后会更新地点资料。
- AI 摘要已生成，请确认后发布。

## 8. 文案、无障碍与数据可信度

### 文案

LIAN 文案气质：清楚、友好、不装、不官方腔、不制造压力。

错误文案必须包含原因或下一步。

不要写：提交失败。

要写：发布失败，网络可能不稳定。你可以重试，草稿已保存。

### 无障碍

- 按钮点击区域不小于 44px。
- 重要文字不能低对比。
- 图标按钮必须有 aria-label。
- 不能只靠颜色区分状态。
- 地图 marker 需要文字或类型图标。
- 动效支持减少动态效果。
- 表单错误必须文字提示。

### 数据可信度

每条可沉淀信息都应能回答：

- 谁提供的？
- 什么时候更新的？
- 是否被确认？
- 是否来自 AI？
- 是否过期？
- 能不能查看来源？

来源类型：用户发布、用户纠错、运营维护、商家更新、AI 整理、官方来源。

可信状态：官方信息、商家更新、运营维护、用户投稿、用户纠错、AI 整理、已确认、待确认、有争议、已过期。

没有来源的 AI 摘要，只能作为草稿，不能作为事实卡片。

## 9. 页面评审 Checklist

每次新增或修改页面前检查：

1. 主信息是否足够清楚？
2. 是否使用统一卡片结构？
3. 主操作是否明显？
4. 玻璃 UI 是否影响阅读？
5. 是否显示当前身份？
6. 是否使用标签 chip？
7. 是否绑定地点？
8. 是否能沉淀到地点页或资料库？
9. AI 内容是否标注来源和不确定性？
10. 错误状态是否告诉用户下一步？
11. 状态是否不只依赖颜色？
12. 触控区域是否足够？
13. 是否支持减少动态效果？
14. 是否符合移动端优先？

## 10. 代码落地约定

V0.1 不迁移框架。当前前端保持静态 classic-script 架构。

落地顺序：

1. 建立 design tokens。
2. 抽象卡片、chip、badge、button、glass 的 CSS 规则。
3. 统一信息流卡片结构。
4. 统一 IdentityBadge / TagChip / LocationChip。
5. 建立状态与反馈模板。
6. 后续再评估组件化或框架迁移。
